# Context — MTConnect SmartSaw

## Glossary

### Install Path
The first-time deployment of the SmartSaw MTConnect stack onto an IPC. Executed by `ssInstall.sh`. Responsible for creating directories under `/etc`, copying configuration from the repo, injecting the serial number into device and devctl configs, and starting Docker Compose.

### Upgrade Path
The update of an already-deployed SmartSaw MTConnect stack. Executed by `ssUpgrade.sh`. Responsible for selectively replacing changed files, pulling updated Docker images, and restarting containers. Must preserve persisted data (MongoDB database, ODS runtime state, adapter logs).

### Selective Upgrade
An upgrade that targets only a subset of components (e.g., `-d` for Agent device file, `-H` for adapter, `-O` for ODS) rather than the full stack with `-A`.

### Full Upgrade
An upgrade that replaces all components and their configurations simultaneously. Triggered by the `-A` flag in `ssUpgrade.sh`.

### Fallback
When `ssUpgrade.sh` detects that `/etc/mtconnect/config/agent.cfg` is missing, it assumes the system was never installed and delegates execution to `ssInstall.sh`.

### Component
One of the six deployable units: **Adapter**, **Agent**, **MQTT Broker**, **ODS**, **Devctl**, **MongoDB**. Each has its own config directory under `/etc` and its own Docker container.

### Data Destruction
The deliberate removal of runtime or persisted state during install/upgrade. In the current codebase, `ssInstall.sh` removes `/etc/mongodb/data/db` and `/etc/ods/config` unconditionally.

### Configuration Drift
When the files in the deployment repo differ from the files currently deployed under `/etc`. Detected by `files_differ()` and `dir_needs_update()` in `lib.sh`.

### Serial Number
The machine-specific identifier that is injected into the MTConnect device XML (`uuid`) and the Devctl JSON config (`device_uid`). Changing it triggers a full component update.

### Idempotency
The property that running the same script multiple times produces the same end state without side effects such as data loss or duplicate processing.

### Graceful Degradation
Continuing to operate in a reduced capacity when part of the system is missing or partially configured, rather than failing hard or destroying state.

### Repository Files
Static assets shipped in the git repo and copied onto the host during Install or Upgrade: configuration files, JSON alarm definitions, device XMLs, Ruby scripts, Python data scripts, CSVs. These are safe to overwrite or remove and replace.

### Runtime State
Files generated by running containers and persisted to the host. Example: MongoDB `/data/db` files stored in `/etc/mongodb/data/db/`. Runtime state must never be destroyed by an Install or Upgrade operation.

### Upgrade Lock
An exclusive advisory lock (`flock`) on `/var/lock/HEMsaw-mtconnect.lock` that ensures only one install or upgrade process runs at a time, protecting `/etc/*` directories and Docker Compose state from concurrent modification via CLI or the IPC Dashboard.
